# 2. 最大隐患：MILP 整袋约束的单位需要重写

你现在写的是：

> $x_j = n_j \times B_j$，其中 $x_j$ 是 kg/t，$B_j$ 是袋重 kg。

这个公式**只有在钢水重量刚好等于 1 吨时才成立**。实际现场一炉可能是 80t、100t、132.2t 或其他重量。你前面参考界面里就有“出钢量 132.2t”这种输入。

正确写法应该是：

$$
x_j \times W = n_j \times B_j
$$

即：

$$
x_j = \frac{n_j \times B_j}{W}
$$

其中：

| 符号 | 含义 |
|---|---|
| $x_j$ | 单耗，kg/t |
| $W$ | 钢水量，t |
| $B_j$ | 第 j 种合金单袋/单批重量，kg |
| $n_j$ | 投料袋数/批数，整数 |

因此，配置里不建议只放 `steel_weight_kg: 1000`。应该改成：

```json
{
  "heat_weight_t": 132.2,
  "calculation_basis": "per_heat"
}
```

输出时同时给：

| 输出 | 单位 |
|---|---|
| 单耗 | kg/t |
| 炉次加入量 | kg/炉 |
| 袋数/批数 | bags/batches |
| 吨钢成本 | 元/t |
| 炉次合金成本 | 元/炉 |

这是必须修正的第一项。

# 3. 第二个隐患：MILP 成本不一定严格小于规则基线

你的 Success Criteria 写了：

> LP/MILP 成本严格 ≤ 规则基线。

这个结论对 LP 基本成立，但对 MILP 不一定成立。原因是：

- LP 是连续变量，理论最低；
- 规则基线如果也是连续浮点，可能没有整袋约束；
- MILP 有整袋/整批限制，约束更严格；
- 因此 MILP 成本可能高于连续规则基线，但仍然更贴近现场。

更严谨的判断应改成：

| 对比 | 正确标准 |
|---|---|
| LP vs 可行规则基线 | LP 成本 ≤ 同约束下的规则基线 |
| MILP vs LP | MILP 成本 ≥ LP，且差距应在合理范围 |
| MILP vs 人工整袋方案 | MILP 应 ≤ 人工整袋方案 |
| MILP vs 规则浮点方案 | 不要求一定更低 |

建议把 Success Criteria 改成：

> LP 是理论下限；MILP 是现场可执行最优；MILP 不要求低于连续规则方案，但应低于或接近人工整袋方案。

# 4. 第三个隐患：成本最低会天然“贴下限”

你的模型目标是最小化合金成本，约束是满足元素上下限。这样求解器通常会把 Mn、Si、Cr 压到有效下限附近，把 C 压到有效上限附近。数学上这是最优，但现场上有风险。

例如：

| 元素 | 纯成本最优倾向 | 现场风险 |
|---|---|---|
| Mn | 靠近下限 | 后续损失后可能低于内控 |
| Si | 靠近下限或被硅锰带高 | 命中率波动 |
| Cr | 靠近下限 | 取样误差后不合格 |
| C | 靠近上限 | 低碳钢超碳风险 |

你现在用了安全余量，这很好，但还不够。建议增加一个“目标偏离惩罚”，让系统不要只压下限，而是在成本和命中稳定性之间折中。

推荐目标函数改成：

$$
\min Z = \sum_j P_j x_j + \lambda_1 \sum_e |C_e^{final}-C_e^{aim}| + \lambda_2 \sum_j y_j
$$

其中：

| 项 | 含义 |
|---|---|
| $\sum P_jx_j$ | 合金成本 |
| $|C_e^{final}-C_e^{aim}|$ | 偏离目标中心的惩罚 |
| $y_j$ | 是否使用某种合金，减少方案过碎 |
| $\lambda_1,\lambda_2$ | 权重 |

V1 可以先不做复杂目标函数，但至少要输出：

> “该方案靠近下限 / 靠近上限 / 风险较高”。

# 5. 第四个隐患：安全余量机制需要从“好/坏元素”改成“双边安全边界”

你现在写的是：

| 类型 | 处理 |
|---|---|
| 有益元素 Mn/Si/Cr | 抬高下限 |
| 有害元素 C/P/S | 压低上限 |

这个思路对 V1 可以用，但现场上不够严谨。因为 Mn、Si、Cr 虽然是有益元素，但也不能超上限；C 对某些钢种也可能有下限要求，不只是有害元素。

建议改成每个元素单独配置：

```json
"safety_margins": {
  "C":  {"low": 0.000, "high": 0.005},
  "Si": {"low": 0.010, "high": 0.010},
  "Mn": {"low": 0.010, "high": 0.010},
  "Cr": {"low": 0.005, "high": 0.005},
  "P":  {"low": 0.000, "high": 0.002},
  "S":  {"low": 0.000, "high": 0.002}
}
```

有效约束统一写成：

$$
L_e + m_e^{low} \le C_e^{final} \le U_e - m_e^{high}
$$

这样比“有益/有害”分类更稳。

# 6. 第五个隐患：P/S 默认值不能让系统给出“虚假安全感”

你方案里已经把 P/S 纳入硬约束，这是非常好的补强。PPT 最后一页只给 C/Si/Mn/Cr，P/S 没有完整展开，而你补充了 P/S，这对现场更安全。`glimmering-foraging-fox.md`

但风险是：你现在写“PPT 未给出，我使用行业经典默认值”。这在开发测试可以接受，但在现场决策不安全。

建议：

| 场景 | 处理 |
|---|---|
| 合金有实际化验单 | 使用实际批次 P/S |
| 没有批次数据但有供应商质保书 | 使用质保书最大值 |
| 只有经验默认值 | 系统必须显示黄色警告 |

# 7. 第六个隐患：温降公式可能严重低估

你现在写：

$$
\Delta T \approx \sum_j(x_j \times 6 / 1000)
$$

并举例总合金量约 33 kg/t，温降约 0.2°C。

这个结果明显偏低。现场上 33 kg/t 合金加入量不太可能只降 0.2°C。温降受合金种类、粒度、加入温度、熔化潜热、钢水量、包况影响，不能简单用这个式子给出非常小的数字。

建议 V1 改成：

1. **默认不输出温降数值**，只输出总合金量 kg/t；
2. 或者输出“粗略估算，不参与决策”；
3. 公式配置化，不写死；
4. 不同合金设置不同降温系数，例如：

```json
"temperature_drop": {
  "enabled": false,
  "coeff_c_per_kg_per_t": {
    "硅锰": 0.6,
    "硅铁": 0.8,
    "高碳锰铁": 0.5,
    "低碳铬铁": 0.7
  }
}
```

如果没有现场校验数据，我建议 V1 暂时只显示：

> 总合金加入量：xx kg/t；温降估算未启用。

> 在 C 上限约束下，高碳合金加入量会被压到 0 或很小。

除非钢种规则明确禁止，否则不应写成自动排除。
