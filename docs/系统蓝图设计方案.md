## 0. 已确认的技术决策

| # | 决策项 | 结论 |

| 1 | 存储 | **PostgreSQL**。JSONB 存规则动作/快照；`percentile_cont` 窗口函数做分位统计；SQLAlchemy 2.x + Alembic；JSON 类型用跨库兼容写法，保留平移 MySQL 的能力；测试用 SQLite 内存跑 |

## 一、系统定位升级

```
现在：给定配置 → 算 rule/lp/milp 三方案
未来：成品钢种/订单 → 映射唯一炼钢牌号 → 内控目标 + 场景约束 + 规则包
      → 多方案优化（成本最低 / 低Si窗口 / 兜底 / 少料种 / 鲁棒）
      → 成本效益分析（宽厚分档热力图）→ 销售快速评估
```

升级后的定位（蓝图原文）：**以炼钢牌号为核心的合金成本优化与牌号效益决策平台**（B/S 多用户）。

四条业务主线：

1. 牌号成分与合金优化（核心计算）
2. 现场规则包（禁硅、锰兜底等经验产品化，插拔即用）
3. 宽厚分档效益（牌号 × 厚度 × 宽度成本/效益热力图）
4. 销售评估报价（钢种 → 牌号 → 成本 → 快速辅助报价）

## 三、目标架构

```
浏览器（Vue3 ESM 无构建 + ECharts，静态资源本地化 static/vendor/）
  ├─ 计算工作台（现 prototype.html 演进：接牌号选择器 + 多方案对比）
  ├─ 主数据管理（牌号/映射/内控目标/合金/成分版本/价格/回收率/终点）
  ├─ 规则包管理（包/规则/范围/冲突检查/执行日志）
  ├─ 批量计算中心（后台任务 + 进度 + 历史结果，替代同步阻塞）
  ├─ 分析看板（宽厚分档热力图/牌号效益分布）
  └─ 销售快速评估
        │ REST
FastAPI 路由层（app/main.py 拆分为 app/routers/）
        │
服务层 app/services/（主数据/规则/场景编排/同步/分析/销售/批量任务）
        │                                    │
核心计算（core.py + rules_engine.py + solvers/）      PostgreSQL
```

**关键原则：计算内核保持无状态纯函数**——输入 config dict → 输出结果，不碰 DB；服务层负责从 DB 组装 config 输入。由此现有 `/api/optimize` 契约与全部测试零破坏。

批量计算改造：FastAPI `BackgroundTasks` + DB 任务状态表（KISS，不引入 Celery/Redis；负载上来再升级）。

### 域 C · 优化求解

| 编号 | 功能 | 类型 | 说明 | 确认 |
|---|---|---|---|---|
| C1 | 目标类型建模 | 〔部〕 | 三种 target_type 编译进 compiled_bounds + 偏好方向，走规则引擎统一入口 | ☐ |
| C2 | 多方案场景编排器 | 〔新〕 | 一次请求产出多方案，见下方详述 | ☐ |
| C3 | 炉重三模式 | 〔部〕 | 吨钢理论模式（默认，无炉重无取整）/ 参考炉重 / 生产投料模式（炉重 + 整袋 + 最小投料量）。现在永远有炉重——改为可选 | ☐ |
| C4 | 最小投料量半连续约束 | 〔新〕 | `min_j · y_j ≤ x_j ≤ max_j · y_j`，仅生产投料模式启用（蓝图明确：吨钢模式下无意义） | ☐ |
| C5 | 鲁棒 MILP | 〔新〕 | 三值参数 + α 等级（R0=0 / R1=0.33 / R2=0.67 / R3=1.0）；下限约束用 worst_low 系数、上限用 worst_high；输出 Final_nominal / worst_low / worst_high + 安全距离 + 风险评级 + 鲁棒性成本。**数据结构先建，单值退化（nominal=low=high）完全向后兼容；启用时点由三值数据可得性决定** | ☐ |
| C6 | 元素集合统一 | 〔部〕 | 核心 6 元素（C,Si,Mn,Cr,P,S）→ 批量模板 17 元素全集（C,Si,Mn,P,S,V,Nb,Ti,Als,Alt,Ca,Cr,Ni,Cu,Mo,B,Sb），单算/批量同一套 | ☐ |
| C7 | 三方案保留 | 〔迁〕 | rule/lp/milp 语义与 `/api/optimize` 契约不变 | ☐ |

**C2 多方案场景编排器详述**（一次请求产出）：

1. **成本最低方案**（基线 C*）
2. **成本窗口方案**：Cost ≤ C* + δ（δ 默认 0.5/1.0 元/t 可配），**词典序两阶段求解**：阶段 1 求 C* → 阶段 2 加成本帽约束行、目标函数切换为 minimize Σ(元素系数 · x)（如低 Si 场景 minimize Si 增量）
3. **禁硅源方案**（规则触发场景）
4. **金属锰兜底方案**：**方式 B（惩罚项）+ 方式 C（两段式：常规无解再启用 + 标记兜底 + 解释）** 组合（蓝图推荐）
5. **少料种方案**（Σy_j ≤ N，N 默认 6）

求解器层扩展：`LinearModel` 支持目标向量替换 / 成本帽约束行 / 二元启用变量；solver 适配层保持"笨"（只翻译模型，不含业务）。

## 六、关键设计原则（Never-break 清单）

1. `/api/optimize` 请求/响应契约不变；rule/lp/milp 语义不变
2. config.json 继续可用；DB 无数据时单机模式照常跑
3. 批量 Excel 模板流程不变（列只增不减）
4. 现有全部测试持续全绿（pytest + node --test）
5. 规则引擎**扩展不重写**：数据驱动规则 → 适配器 → 插件实例注入现有 RuleEngine
6. 计算内核无状态纯函数，DB 只在服务层出现
7. 手工铝双轨语义不变（手工铝不计入最终成本合计口径）

## 八、依赖关系（非排期，仅结构）

```
G1 数据库 ──┬─ A 主数据 ──┬─ G2 外部同步
            │             ├─ B 规则包
```
